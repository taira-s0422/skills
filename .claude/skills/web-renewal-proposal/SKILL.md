# Webリニューアル提案スキル

Webリニューアルの提案プロセスを自動化し、マーケティング視点を統合した最適な提案資料を作成するためのスキルです。ユーザー（クライアント）の潜在的なニーズを言語化し、データに基づいた戦略的な提案を構築する体系的なワークフローを提供します。

---

## The Iron Laws (絶対法則)

```
1. 分析なくして提案なし (NO PROPOSAL WITHOUT ANALYSIS FIRST)
2. ヒアリングなくして顧客なし (NO PERSONA WITHOUT HEARING FIRST)
3. 目的なくして手段なし (NO SOLUTION WITHOUT GOAL FIRST)
```

これらの法則に違反することは、提案の質を損なうことを意味します。

---

## 使用タイミング

以下のようなリクエストがあった場合、このスキルを起動します：

- 「Webリニューアルの提案書を作成したい」
- 「クライアントのサイト改善提案を作りたい」
- 「Webサイトの課題分析から提案までをまとめたい」
- 「マーケティング視点でのサイトリニューアル戦略を立てたい」
- `/web-renewal-proposal`

## 提案ワークフロー (5フェーズ)

### フェーズ1: ゴール設定と現状分析 (Discovery Gate)

**目的:** プロジェクトの全体像を把握し、リニューアルの目的（KGI）と現状の課題を客観的なデータに基づいて明確化します。

#### ステップ1.1: 背景と目的の定義

以下の質問を**一つずつ**順番に聞きます：

1. 「なぜWebサイトのリニューアルが必要だと感じていますか？」
2. 「リニューアルによって、最終的に達成したいビジネス目標（KGI）は何ですか？」
   - 例：「Web経由の売上を昨年比120%にする」「月間問い合わせ件数を10件から30件に増やす」

**注意:** 「デザインを新しくしたい」などの手段ベースの回答が出た場合、「それによって、どのようなビジネス成果を期待していますか？」と再度問いかけ、目的を明確にします。

#### ステップ1.2: 内部環境分析

以下の質問を**一つずつ**順番に聞きます：

1. 「現在のWebサイトは、どのような役割を担っていますか？」
2. 「営業担当者や採用担当者など、社内の関係者は現在のサイトについてどのような課題を感じていますか？」

**ヒアリング先の例:**
- 営業部：「顧客に見せられない」「説明資料が不足」
- 広報部：「更新に時間がかかる」「情報発信が遅れる」
- 採用担当：「応募者が集まらない」「会社の魅力が伝わらない」

#### ステップ1.3: 外部環境分析

以下の質問を**一つずつ**順番に聞きます：

1. 「主要な競合企業（2〜3社）を教えてください」
2. 「それらの競合企業のWebサイトの強み・弱みは何だと思いますか？」

**分析のポイント:**
- 競合サイトを実際にブラウザで確認し、デザイン、コンテンツ、機能を比較
- 差別化ポイントを特定

#### ステップ1.4: データ分析

以下のデータを収集・分析します：

| 分析項目 | 確認方法 | 課題の判断基準 |
|---------|---------|---------------|
| 直帰率 | Google Analytics | 70%以上は高い |
| CVR | Google Analytics | 業界平均と比較 |
| 流入経路 | Google Analytics | 指名検索のみは問題 |
| デバイス比率 | Google Analytics | モバイル比率が高いのにスマホ未対応は機会損失 |

**ゲートチェック:**

```
✅ 以下がすべて揃っているか確認：
- [ ] KGIが明確に定義されている
- [ ] 社内関係者の課題が収集されている
- [ ] 競合分析が完了している
- [ ] 定量データ（GA等）が分析されている

すべて揃っていない場合、フェーズ2に進まず、不足情報を収集します。
```

---

### フェーズ2: 顧客理解とペルソナ設定 (Persona Gate)

**目的:** 誰に情報を届け、どのような行動を促したいのかを定義します。潜在ニーズを引き出すためのヒアリングを体系的に行い、具体的なペルソナを構築します。

#### ステップ2.1: ターゲット定義

以下の質問を聞きます：

「Webサイトで情報を届けたい主なターゲット顧客は誰ですか？」
- BtoB: 企業規模、業種、担当者の役職
- BtoC: 年齢層、性別、ライフスタイル

#### ステップ2.2: 潜在ニーズ深掘り

以下の4つの質問テクニックを**順番に**使い、潜在ニーズを引き出します：

**1. オープンな質問**（顧客の自由な回答を引き出す）
- 例：「最近、〇〇（業務/生活）において、何かお困りのことはありますか？」

**2. 具体的な質問**（具体的な状況やエピソードを尋ねる）
- 例：「『〇〇が困る』とおっしゃいましたが、具体的にはどのような時ですか？」

**3. なぜ・どうして質問**（顧客の思考や感情を深掘りする）
- 例：「なぜそのように感じたのですか？」「その時、どのような気持ちでしたか？」

**4. 仮説を提示する質問**（顧客の潜在的なニーズを刺激する）
- 例：「もし、〇〇という機能があれば、その課題は解決しそうですか？」

**心理的テクニック:**
- **共感:** 「それは大変でしたね」と相手の感情に寄り添う
- **傾聴:** 繰り返し出てくるキーワードに注目する
- **沈黙:** 質問後、少し間を置き、相手に考える時間を与える

#### ステップ2.3: ペルソナ作成

収集した情報を基に、以下の形式でペルソナを作成します：

```
👤 PERSONA: [具体的な名前]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## プロフィール
- **役職/属性:** [具体的な役職または属性]
- **企業/背景:** [企業規模、業種、家族構成など]
- **年齢:** [年齢層]

## 課題 (Pain Points) - 優先度順
1. [最も深刻な課題 - 具体的に]
2. [2番目の課題]
3. [3番目の課題]

## 目標 (Goals)
1. [達成したい主な目標]
2. [副次的な目標]

## 情報収集方法
- [どのように情報を探すか]
- [信頼する情報源]

## 潜在ニーズ
[ヒアリングで明らかになった、本人も気づいていなかった深層のニーズ]

**エビデンスソース:** [この情報の出所]
```

**ペルソナ数の制限:**
- 最大3名まで
- 3名以上になる場合、「どのペルソナが最も売上に貢献しますか？」と問いかけ、優先順位をつける

**ゲートチェック:**

```
✅ 以下がすべて揃っているか確認：
- [ ] ターゲット顧客が具体的に定義されている
- [ ] 潜在ニーズが言語化されている
- [ ] ペルソナが作成されている（最大3名）
- [ ] エビデンスソースが明記されている

すべて揃っていない場合、フェーズ3に進まず、不足情報を収集します。
```

### フェーズ3: 解決策と戦略の策定 (Strategy Gate)

**目的:** 分析結果とペルソナに基づき、課題を解決するための具体的な施策と目標（KPI）を策定します。

#### ステップ3.1: 施策のブレインストーミング

フェーズ1で洗い出した課題と、フェーズ2で明らかになった潜在ニーズを基に、解決策をリストアップします。

| 課題 / ニーズ | 解決策のアイデア |
|-------------|-----------------|
| [課題1] | [施策案1]、[施策案2] |
| [潜在ニーズ] | [それに応える施策案] |

#### ステップ3.2: 施策の具体化とKPI設定

各施策について、以下の表形式で具体化します：

| 施策内容 | 期待効果 | 達成すべき目標 (KPI) |
|---------|---------|---------------------|
| [具体的な施策] | [どのように課題を解決するか] | [測定可能な数値目標] |

**KPI設定の原則:**
- 測定可能な数値であること
- 現状値と目標値を明示すること
- 努力すれば達成可能なラインであること

**例:**

| 施策内容 | 期待効果 | 達成すべき目標 (KPI) |
|---------|---------|---------------------|
| レスポンシブデザイン導入 | スマホユーザーの離脱を防ぐ | スマホ経由の直帰率を85% → 50%以下に改善 |
| 「導入事例」ページの作成 | 顧客の信頼を獲得 | 公開後半年で月間500PV、問い合わせユーザーの80%が閲覧 |

#### ステップ3.3: サイトマップ設計

新しい施策を盛り込んだサイト構造を設計します。

**ゲートチェック:**

```
✅ 以下がすべて揃っているか確認：
- [ ] 各課題に対する解決策が提示されている
- [ ] 各施策に測定可能なKPIが設定されている
- [ ] サイトマップが設計されている
- [ ] 施策がペルソナのニーズに応えている

すべて揃っていない場合、フェーズ4に進まず、不足情報を収集します。
```

---

### フェーズ4: 提案書の作成 (Proposal Gate)

**目的:** これまでのフェーズで得られた情報を統合し、決裁者の承認を得るための説得力のある提案書を作成します。

#### 提案書の構成（8項目）

1. **サイトリニューアルの概要** (サマリー)
2. **リニューアルの背景と目的 (KGI)**
3. **現状の課題分析 (自社・競合・データ)**
4. **ターゲット設定とペルソナ**
5. **施策内容とリニューアル後の目標 (KPI)**
6. **実装・運用に関する検討項目**
7. **予算・スケジュール**
8. **制作体制と見積もり比較**

#### 各セクションの書き方

**1. サイトリニューアルの概要**

A4用紙1枚程度で以下をまとめます：
- リニューアルの目的（KGI）
- 現状の課題（3つ程度）
- 中心となる施策
- 概算予算
- 期間
- 期待される効果

**2. リニューアルの背景と目的 (KGI)**

- **背景:** 経営課題や事業環境の変化を記載
- **目的:** 具体的な数値目標（KGI）を明記

**注意:** 「デザインが古いから」ではなく、「ビジネス課題」を中心に語る

**3. 現状の課題分析**

- **定量データ:** Google Analyticsなどの数値を提示
- **定性データ:** 社内関係者の声を引用
- **競合比較:** 競合の強みと自社の弱みを対比

**4. ターゲット設定とペルソナ**

フェーズ2で作成したペルソナをそのまま掲載

**5. 施策内容とリニューアル後の目標 (KPI)**

フェーズ3で作成した施策とKPIの表を掲載

**6. 実装・運用に関する検討項目**

- CMS（更新システム）の選定方針
- 更新・運用体制の明確化
- セキュリティ要件（常時SSL、フォームセキュリティなど）
- SEO・アクセス解析の導入

**7. 予算・スケジュール**

| 項目 | 金額（概算） | 備考 |
|-----|------------|------|
| **初期費用（制作費）** | 〇〇万円 | デザイン、コーディング、CMS構築、システム開発費を含む |
| **月額費用（保守・運用）** | 〇万円 | サーバー・ドメイン代、保守管理費、更新代行費など |

- **スケジュール:** 承認後、約〇ヶ月で公開予定（バッファを含む）

**8. 制作体制と見積もり比較**

- プロジェクトメンバー（社内・外部）の役割分担
- 複数社から取得した見積もりの比較（可能であれば）

**ゲートチェック:**

```
✅ 以下がすべて揃っているか確認：
- [ ] 8つの構成要素がすべて含まれている
- [ ] 「ビジネス成果」が中心に据えられている
- [ ] 専門用語には注釈がついている
- [ ] 定量データが根拠として示されている

すべて揃っていない場合、フェーズ5に進まず、修正します。
```

---

### フェーズ5: 最終確認と納品 (Verification Gate)

**目的:** 提案書全体の品質を最終確認し、ユーザーに納品します。

#### 検証チェックリスト

提案書を最終納品する前に、以下の項目をすべて満たしていることを確認します：

- [ ] **絶対法則の遵守:** 3つの絶対法則に違反していないか？
- [ ] **データソースの明記:** すべての分析や主張に根拠となるデータが示されているか？
- [ ] **一貫性:** 課題、ペルソナ、施策、KGI/KPIがすべて一貫したストーリーになっているか？
- [ ] **専門用語の排除:** 決裁者が理解できる平易な言葉で書かれているか？
- [ ] **ビジネス成果の強調:** デザインの美しさではなく、ビジネスへの貢献が中心に据えられているか？
- [ ] **現実的な計画:** 予算とスケジュールは現実的で、リスクが考慮されているか？

すべてクリアした場合、提案書（Markdown形式）をユーザーに納品します。

## 品質管理 (Quality Control)

### 品質基準 (Quality Gates)

| 要素 | 良い例 (Good) | 悪い例 (Bad) - 却下 |
|------|--------------|-------------------|
| **目的 (KGI)** | 「Web経由の売上を昨年比120%にする」 | 「サイトをきれいにしたい」 |
| **課題** | 「スマホでの直帰率が90%を超えている」 | 「なんとなく使いにくい」 |
| **ターゲット** | 「40代の生産管理部長。コスト削減に課題」 | 「すべての中小企業」 |
| **施策** | 「CMSを導入し、更新外注費をゼロにする」 | 「コンテンツを充実させる」 |

### 警告フラグ (Red Flags) - STOP

以下の表現やパターンが出現した場合、プロセスを一時停止し、軌道修正を促します：

| パターン | 対処法 |
|---------|--------|
| **「デザインの好み」に関する議論** | 「このデザインは、ペルソナである〇〇さんの課題解決にどう繋がりますか？」と問いかけ、議論を戦略ベースに戻す |
| **根拠のない断定** | 「その予測の根拠となるデータや事例はありますか？」と問いかけ、客観性を求める |
| **手段の目的化** | 「〇〇を導入することで、どのビジネス課題を解決しますか？」と問いかけ、目的を再確認させる |
| **「革新的」「最先端」などの空虚な表現** | 「具体的には何が革新的なのですか？」と問いかけ、具体性を求める |
